#+title: overleaf-project（中文说明）
#+subtitle: Emacs 中基于 Git 的 Overleaf 项目同步工具

[[https://github.com/Jamie-Cui/overleaf-project/actions/workflows/test.yml][https://github.com/Jamie-Cui/overleaf-project/actions/workflows/test.yml/badge.svg]]
[[https://github.com/Jamie-Cui/overleaf-project/actions/workflows/coverage.yml][https://github.com/Jamie-Cui/overleaf-project/actions/workflows/coverage.yml/badge.svg]]

English README: [[file:README.org][README.org]]

~overleaf-project~ 在项目层级集成 Emacs 与 Overleaf。它会将 Overleaf 项目克隆
为本地 Git 仓库，在仓库中记录 Overleaf 项目元数据，并在本地 Git 历史与
Overleaf 之间同步完整项目快照。

本包面向希望使用本地 LaTeX 编辑、普通 Git 提交和 Git 冲突处理，同时仍需
与使用 Overleaf Web 编辑器的协作者保持同步的用户。

* 功能

- 将完整 Overleaf 项目克隆为本地 Git 仓库。
- 将已有 Git 仓库绑定到 Overleaf 项目。
- 将本地 Git 快照推送到 Overleaf，并把远端 Overleaf 改动拉回当前分支。
- 通过比较上一次同步的 commit、本地 ~HEAD~ 和最新 Overleaf 快照检测本地
  与远端是否分叉。
- 使用普通 Git 合并工具处理冲突，包括 Magit。
- 在可行时通过 Overleaf 实时文本 OT 通道更新已有文本文档，从而保留远端
  Overleaf 文档 id。
- 将认证 cookies 存入 ~auth-source~、普通 cookie 文件，或仅保存在当前
  Emacs 会话中。
- 可选地在后台 Emacs 线程中执行网络、解压和 Git 操作。
- 为 Overleaf 管理的仓库提供可选的 ~magit-status~ 集成。

~overleaf-project~ 不提供 [[https://github.com/vale981/overleaf.el][overleaf.el]] 的单 buffer 实时编辑工作流。新配置和
既有配置都应直接加载 ~overleaf-project~。

* 需求

~overleaf-project~ 要求 Emacs 29.4 或更新版本，并依赖以下 Elisp 包：

- =websocket= 1.15 或更新版本
- =webdriver= 0.1 或更新版本

项目同步还要求系统中可用以下外部可执行文件：

- =git=
- =curl=
- =unzip=

默认认证后端通过 Selenium webdriver 使用 Firefox，因此需要安装
=geckodriver=。Firefox cookie 导入后端不需要 =geckodriver=，但要求当前
Emacs 构建支持 SQLite。

* 安装

** 包归档

如果通过 MELPA 等 package archive 安装，可使用 Emacs 的包管理器：

#+begin_example
M-x package-refresh-contents RET
M-x package-install RET overleaf-project RET
#+end_example

使用 =use-package=：

#+begin_src elisp
  (use-package overleaf-project
    :ensure t
    :custom
    (overleaf-project-url "https://www.overleaf.com"))
#+end_src

如果使用自建 Overleaf 实例，请在使用交互命令前设置服务器地址：

#+begin_src elisp
  (setopt overleaf-project-url "https://latex.example.edu")
#+end_src

** 手动 checkout

如果使用本地 checkout，请单独安装 Elisp 依赖，并将本仓库加入
~load-path~：

#+begin_src elisp
  (add-to-list 'load-path "/path/to/overleaf-project")
  (require 'overleaf-project)
#+end_src

包 feature、交互命令和自定义变量均使用 ~overleaf-project-~ 前缀。

* 快速开始

1. 认证 Overleaf：

   #+begin_example
   M-x overleaf-project-authenticate
   #+end_example

2. 克隆 Overleaf 项目：

   #+begin_example
   M-x overleaf-project-clone
   #+end_example

3. 在本地编辑并使用 Git 提交改动。

4. 将本地提交推送到 Overleaf：

   #+begin_example
   M-x overleaf-project-push
   #+end_example

5. 当协作者在 Web 编辑器中修改远端项目后，拉取 Overleaf 改动：

   #+begin_example
   M-x overleaf-project-pull
   #+end_example

如果项目已经存在于本地 Git 仓库中，请运行 ~overleaf-project-init~，而不是
~overleaf-project-clone~，将该仓库绑定到 Overleaf 项目。

* 认证

clone、push 或 pull 前需要有效的 Overleaf 会话 cookies。主要命令为：

#+begin_example
M-x overleaf-project-authenticate
#+end_example

如果 ~overleaf-project-clone~、~overleaf-project-init~、
~overleaf-project-push~、~overleaf-project-overwrite-remote~ 或
~overleaf-project-pull~ 启动时没有有效 cookies，命令会在 minibuffer 中提示
先运行 ~overleaf-project-authenticate~。

** Webdriver 后端

默认后端为 ~webdriver~：

#+begin_src elisp
  (setopt overleaf-project-auth-backend 'webdriver)
#+end_src

它会在空闲本地端口启动 =geckodriver=，打开 Firefox，并要求你登录
Overleaf。即使有旧的 =geckodriver= 进程占用默认端口 =4444=，也不应阻止
重新认证。

默认情况下，认证后的 cookies 会存入 Emacs ~auth-source~。本包会使用
~auth-sources~ 中的第一个明文文件条目，并在没有可用条目时回退到
=~/.authinfo=。它按 Overleaf host 管理一条记录，并使用
=login overleaf-project= 与 =port overleaf-project-cookie= 进行定位。
重新认证会替换该受管理记录，而不是追加过期副本。

** Firefox cookie 导入

如果 Firefox 已经登录 Overleaf，可以从默认 Firefox profile 导入 cookies：

#+begin_src elisp
  (setopt overleaf-project-auth-backend 'firefox-cookies)
#+end_src

该后端会读取 =profiles.ini=，只导入当前 ~overleaf-project-url~ 对应的
cookies，并在找不到有效 Overleaf session cookie 时给出明确失败信息。若要
跳过 profile 自动发现：

#+begin_src elisp
  (setopt overleaf-project-firefox-profile "/path/to/firefox/profile")
#+end_src

** Cookie 存储

将 cookies 存入普通文件：

#+begin_src elisp
  (setopt overleaf-project-cookie-storage "~/.overleaf-project-cookies")
#+end_src

只在当前 Emacs 会话内存中保留 cookies：

#+begin_src elisp
  (setopt overleaf-project-cookie-storage nil)
#+end_src

高级配置可以直接设置 ~overleaf-project-cookies~ 和
~overleaf-project-save-cookies~。

对于 ~overleaf-project-authenticate~ 保存的 cookies，本包会在任何网络请求
前基于本地保存的 session-cookie 过期时间检查会话是否有效。短生命周期的
分析统计类 cookies 不参与该检查。

* 工作流

** 克隆项目

执行：

#+begin_example
M-x overleaf-project-clone
#+end_example

该命令会提示选择 Overleaf 项目，下载完整项目 zip，创建本地 Git 仓库，提
交导入后的快照，并将 Overleaf 元数据写入仓库的本地 Git 配置。

** 绑定已有仓库

在已有 Git 仓库中执行：

#+begin_example
M-x overleaf-project-init
#+end_example

该命令会提示选择远端 Overleaf 项目，将项目元数据写入本地 Git 配置，并初
始化后续 push 和 pull 使用的隐藏 base snapshot。它不会自动从 Overleaf 拉
取，也不会自动推送到 Overleaf。

** 本地编辑

clone 或 init 后，将该仓库作为普通 Git 工作树使用：

- 在本地编辑文件；
- 按需暂存并提交改动；
- 运行 ~overleaf-project-push~ 上传本地提交；
- 运行 ~overleaf-project-pull~ 合入远端 Overleaf 改动。

~overleaf-project-push~ 会在访问 Overleaf 前准备本地分支。如果存在已暂存
改动，它会自动提交；如果存在未暂存或未跟踪文件，它会询问是否先暂存所有
改动。随后它会获取最新远端快照，并且只在远端未分叉时上传本地 ~HEAD~。

~overleaf-project-pull~ 要求工作树干净，并且必须在普通分支上运行，不能在
detached ~HEAD~ 状态下运行。

** 推送到 Overleaf

执行：

#+begin_example
M-x overleaf-project-push
#+end_example

该命令会比较三种状态：

- 上一次成功同步的 Git commit；
- 当前本地 ~HEAD~；
- 从 Overleaf 下载的最新项目快照。

处理规则如下：

| 状态 | 结果 |
|------+------|
| 只有本地发生变化 | 将本地 ~HEAD~ 上传到 Overleaf。 |
| 远端已经与 ~HEAD~ 一致 | 只更新 base ref。 |
| 只有远端发生变化 | 停止并提示运行 ~overleaf-project-pull~。 |
| 本地与远端都发生了不同变化 | 停止并提示先 pull。 |

对于远端已有的 Overleaf 文本文档，发生变化的 UTF-8 文本会通过 Overleaf 的
实时 ShareJS 文本 OT 路径更新。这样会保留远端文档 id，并在 Overleaf Web
端生成正常的历史记录。发生变化的二进制文件、新增文件、类型冲突和删除条
目会使用 Overleaf 上传和删除 API。

如果某个远端文档无法通过当前支持的文本 OT 更新，例如它不是合法 UTF-8、使
用 Overleaf 较新的 ~history-ot~ 类型，或在下载快照后又被远端修改，push 会
失败，而不会静默删除并重建该文档。

** 从 Overleaf 拉取

执行：

#+begin_example
M-x overleaf-project-pull
#+end_example

该命令会比较同样的三种状态。处理规则如下：

| 状态 | 结果 |
|------+------|
| 只有远端发生变化 | 在本地 fast-forward 当前分支。 |
| 本地已经与远端快照一致 | 只更新 base ref。 |
| 远端没有新变化 | 提示当前无需拉取。 |
| 本地与远端都发生了不同变化 | 将远端快照合并到当前分支。 |

如果因远端分叉产生了干净合并，请随后运行 ~overleaf-project-push~，将合并
结果发布回 Overleaf。

** 覆盖 Overleaf 远端

执行：

#+begin_example
M-x overleaf-project-overwrite-remote
#+end_example

这是一个恢复命令。它使用与 ~overleaf-project-push~ 相同的本地准备流程，但
总是用本地 ~HEAD~ 快照替换远端 Overleaf 项目内容。已有远端文本文档在支持
时仍会通过文本 OT 更新；其他条目使用 Overleaf 上传和删除 API。只有在确实
希望本地状态替换远端状态时才应使用该命令。

** 冲突处理

当本地与远端快照分叉时，先运行 ~overleaf-project-pull~。该命令会使用普通
Git 合并机制，将下载下来的远端快照合并到当前分支。

如果合并产生冲突：

- 当前分支会保持在 Git 的标准冲突合并状态；
- 使用 Magit 或普通 Git 解决冲突；
- 创建 merge commit；
- 运行 ~overleaf-project-push~ 上传合并结果。

冲突处理有意交给 Git，而不是 ediff。

* 同步细节

clone、init、push 和 pull 命令会将项目元数据存入仓库的本地 Git 配置。仓
库绑定后，可以从该仓库内任意目录继续运行 push 和 pull。

当 ~overleaf-project-sync-metadata-enabled~ 为非 nil 时，成功的 push 操作
会在 Overleaf 项目根目录维护 ~.overleaf-project-sync.json~。该文件记录本
包上次上传的本地 Git commit 和 tree。下载快照后，它会在 Git 比较前被移
除，因此它是同步 bookkeeping，而不是普通项目内容。它不应被本地 Git 仓库
跟踪。

在可能移动当前分支或完成 pending sync 状态的同步步骤前，如果
~overleaf-project-local-backups-enabled~ 为非 nil，本包会在
~refs/overleaf-project/backups/~ 下创建本地安全 refs。这些 refs 会让之前
的本地 commit 在分支移动和 reflog 过期之外仍然可达。

查看 backup refs：

#+begin_example
git show-ref refs/overleaf-project/backups
#+end_example

从 backup ref 创建普通分支以恢复：

#+begin_example
git branch recover-overleaf refs/overleaf-project/backups/...
#+end_example

空目录不会被 Git 跟踪，因此 push 或 pull 时也不会被保留。

* 异步操作

默认情况下，命令同步运行。若希望在耗时网络、解压和 Git 工作期间保持
Emacs 响应：

#+begin_src elisp
  (setopt overleaf-project-enable-async t)
#+end_src

交互命令仍会在前台收集 minibuffer 输入和确认，随后在后台执行耗时工作。
这适用于 clone、init、push、pull、远端覆盖、认证和 Magit 远端刷新。

取消当前所有已跟踪的 Overleaf 后台操作：

#+begin_example
M-x overleaf-project-force-stop
#+end_example

取消会尽可能中断外部进程、清理 async 锁，并丢弃尚未执行的前台回调。已
经在本地或远端完成的工作不会自动回滚。

* Magit 集成

若要在 =magit-status= 中显示 Overleaf section，请加载可选集成：

#+begin_src elisp
  (with-eval-after-load 'magit
    (require 'overleaf-project-magit)
    (overleaf-project-magit-setup))
#+end_src

对于 Overleaf 管理的仓库，当 ~overleaf-project-enable-async~ 为非 nil 时，
每次 =magit-status= 刷新也可能在后台刷新远端 Overleaf 快照。自动刷新会被
内部节流，避免普通 Magit 刷新反复下载快照。

若要保持手动远端刷新：

#+begin_src elisp
  (setopt overleaf-project-magit-auto-refresh-remote nil)
#+end_src

仍可手动刷新远端状态：

#+begin_example
M-x overleaf-project-magit-refresh-remote
#+end_example

* Commit Hook

若要在 Emacs 中通过 Magit 或 =git-commit= 创建 commit 后自动 push：

#+begin_src elisp
  (with-eval-after-load 'git-commit
    (add-hook 'git-commit-post-finish-hook
              (lambda () (overleaf-project-push nil t))))
#+end_src

该 hook 只会 push 已经包含 Overleaf 项目元数据的仓库。命令同步运行时，该
hook 会在 push 运行期间阻塞 Emacs。启用 ~overleaf-project-enable-async~
可让 hook 触发的 push 在后台运行。

* 自定义

重要用户选项包括：

| 选项 | 说明 |
|------+------|
| ~overleaf-project-url~ | Overleaf 服务器 URL。 |
| ~overleaf-project-auth-backend~ | 认证后端：~webdriver~ 或 ~firefox-cookies~。 |
| ~overleaf-project-cookie-storage~ | Cookie 存储后端：~authinfo~、文件路径或 nil。 |
| ~overleaf-project-firefox-profile~ | Firefox cookie 导入使用的显式 profile 路径。 |
| ~overleaf-project-enable-async~ | 在后台线程中运行耗时交互操作。 |
| ~overleaf-project-sync-auto-commit-message~ | push 前自动本地 checkpoint commit 的提交信息。 |
| ~overleaf-project-sync-metadata-enabled~ | 启用远端同步元数据文件。 |
| ~overleaf-project-sync-metadata-file~ | 保留的远端元数据文件名。 |
| ~overleaf-project-local-backups-enabled~ | 同步操作期间创建本地安全 refs。 |
| ~overleaf-project-local-backup-ref-prefix~ | 本地安全 refs 使用的 ref namespace。 |
| ~overleaf-project-socket-timeout~ | websocket 获取项目树的超时时间。 |
| ~overleaf-project-git-executable~ | Git 可执行文件路径。 |
| ~overleaf-project-curl-executable~ | Curl 可执行文件路径。 |
| ~overleaf-project-unzip-executable~ | Unzip 可执行文件路径。 |
| ~overleaf-project-debug~ | 启用详细 debug 日志。 |
| ~overleaf-project-log-echo~ | 在 minibuffer 中同时显示日志条目。 |
| ~overleaf-project-magit-auto-refresh-remote~ | 从 Magit 自动刷新远端快照。 |

使用 =M-x customize-group RET overleaf-project RET= 查看完整 customization
group。

* 日志

Overleaf 信息、警告和 debug 条目写入全局
~*overleaf-project-log*~ buffer。每条记录都包含时间戳、日志级别，以及当前
可获得的项目、仓库或 Overleaf URL 上下文，便于区分多个项目的日志。

启用详细 debug 日志：

#+begin_src elisp
  (setopt overleaf-project-debug t)
#+end_src

* 安全说明

Overleaf cookies 可以访问你的 Overleaf 账户。请保护 auth-source 文件和自
定义 cookie 文件，不要提交真实 cookies、项目标识符、ancestor backup 文
件，或包含会话数据的 debug 日志。

* 开发

仓库包含一个简单的 =Makefile= 用于本地验证：

#+begin_example
make
make test
make coverage
make help
make clean
#+end_example

如果 Emacs 可执行文件在 =PATH= 中不是 =emacs=：

#+begin_example
make EMACS=/path/to/Emacs
#+end_example

等价的 batch 验证命令：

#+begin_example
emacs -Q --batch -L . --eval "(progn (require 'package) (package-initialize))" -f batch-byte-compile overleaf-project.el
emacs -Q --batch -L . --eval "(progn (require 'package) (package-initialize))" --eval "(require 'overleaf-project)"
#+end_example

自动化测试套件使用 ERT，并设计为可离线运行：

#+begin_example
make test
#+end_example

它覆盖纯 helper、cookie 和 HTTP 边界逻辑、本地 Git 快照与状态流转、使用
stub Overleaf API 的远端树 diff、命令分发、async 状态处理，以及少量
Magit 集成 helper。它有意不连接 Overleaf、不启动 webdriver，也不打开浏览
器。

对于影响真实服务边界的改动，仍应使用真实 Overleaf 会话验证 clone、init、
push、pull、pending-state recovery 和 Git 冲突处理。对于认证相关改动，请
尽可能验证 webdriver 认证和 Firefox cookie 导入。

当前默认构建不会自动运行覆盖率统计，但仓库提供基于 Emacs 内置
=testcover= 的可选目标：

#+begin_example
make coverage
make coverage COVERAGE_MIN=50
#+end_example

该目标会将 tab 分隔的摘要写入 =coverage/testcover-summary.tsv=。可选的
=COVERAGE_MIN= 变量会把报告变成阈值检查。GitHub Actions 也提供手动
=workflow_dispatch= 触发入口，会运行 =make coverage= 并上传摘要 artifact。
外部包 =undercover.el= 也可以把 ERT 覆盖率上报到托管覆盖率服务，但这不属
于默认 CI 路径。
